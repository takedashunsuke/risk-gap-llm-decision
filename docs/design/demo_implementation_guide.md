# LLMシミュレーター・デモ実装ガイド

提出までの短期間（数日スケール）向けに、`high_level_design_v2.md` と現行リポジトリの関係、実装手順、記録方式の提案をまとめる。

---

## 1. 現状の整理：設計書とコードベースのギャップ

| 項目 | `high_level_design_v2.md` | 現行コード（`main.py` / `simulation.py` など） |
|------|---------------------------|-----------------------------------------------|
| 世界観 | 単一シナリオ（登山・レジャー等）、**状態変数→R_obj/R_subj/Gap** | **2Dマルチプレイス＋火災**、エージェント移動・通信 |
| 意思決定 | **線形結合＋閾値 T・Cost_stop** で続行/中止 | LLMが JSON で **move/stay** を出力 |
| LLMの役割 | 客観リスク提示・Gap指摘・**中止提案（装置）** | エージェントの**発話・行動**の生成 |
| デモ要件 | 1.5〜3分で **数値ダッシュボード＋分岐比較** | Matplotlib / `output` 画像・jsonl ログ |

**結論（短期デモ）:** 設計書どおりの「構造可視化デモ」は、既存の2Dシミュレーションをそのまま押し込むより、**軽量な専用デモ（数値モデル＋最小UI）** を別経路で用意するのが速い。既存シミュは「技術スタックの裏付け・拡張」として資料で触れるとよい。

本リポジトリでは **`apps/demo_web/`** に、設計書セクション4〜7に沿った**簡易状態機械＋Web UI** を置く（`localhost` で起動）。

---

## 2. デモの完成条件（設計書「完成条件」そのまま）

1. **リスクが上がる様子が見える** → R_obj・内訳（環境/人/時間）の推移表示  
2. **人が間違える理由が見える** → Gap・バイアス・Cost_stop の表示  
3. **介入で結果が変わる** → 「続行→事故」と「LLM介入→中止→回避」の2分岐  

プレゼン構成（`docs/presentation/v1.md`）のスライド⑧と整合させ、**左＝事故／右＝回避** の比較が一目で分かるレイアウトにする。

## 3. 実装手順（推奨順・3日以内向け）

### フェーズ A：コアモデル（半日〜1日）

1. **状態変数**（設計書 4.1）を dict または dataclass で保持  
   - Environment（天候・視界・気温リスク）  
   - Human（疲労・注意力低下）  
   - Pressure（時間・外的圧力）  
2. **R_obj**（設計書 4.2）：`w1*環境平均 + w2*人間平均 + w3*時間`（初期は等重み 0.33 で可）  
3. **バイアス**（設計書 4.3）：経験・圧力・損失回避を **簡易スカラー** に畳む（成功体験 +0.1 など）  
4. **R_subj = R_obj - バイアス**、**Gap = R_obj - R_subj**  
5. **Cost_stop**（4.4）：時間損失・目標未達・社会的圧力を 0〜1 に正規化したスカラー  
6. **分岐**（4.5）：設計書の式  
   - `続行 if (R_subj - Cost_stop) < T`  
   - それ以外は中止  

※短期間は **決定論的なステップ進行**（シナリオで各ステップの悪化量を固定）にすると、デモの再現性が高い。

### フェーズ B：UI（半日〜1日）

1. **常時表示**（設計書 8）：R_obj, R_subj, Gap, T, Cost_stop, リスク内訳  
2. **操作**：ステップ進行、「続行（過信）」「LLM介入して中止」のボタン  
3. （任意）LLMは **本文生成のみ**：Ollama に「現在の数値と Gap を踏まえた短い警告文」を依頼し、テキストとして表示。**数値の真実は Python 側**に固定し、ハルシネーションで数値が崩れないようにする。

### フェーズ C：発表用の仕上げ（数時間）

1. タイムラインを **90〜180秒** に収める（デモ UI はステップ上限を **10〜100** で変更可能。**自動再生**で早送りするか、発表用は上限を **20〜40** 程度にすると操作が軽い）  
2. スライド⑧と同じ **数値例**（例：R_obj 0.72, R_subj 0.45, Gap 0.27）が出るまでパラメータを微調整  
3. 動画録画用にブラウザウィンドウサイズを固定

## 4. ローカルでの起動方法（`apps/demo_web`）

ホストで Python を使う場合（Ollama は従来どおりホストで可）:

```bash
cd /path/to/2d-multi-places-simulation-on-fire-public
pip install -r apps/demo_web/requirements.txt
cd apps/demo_web
uvicorn app:app --host 127.0.0.1 --port 8765
```

ブラウザで `http://127.0.0.1:8765` を開く。

Docker で動かす場合は **`docker/docker-compose.yml`** の **`web`** サービスのみ。`OLLAMA_BASE_URL` は `host.docker.internal` を指す。手順は **`docs/CLAUDE.md`** の「Docker（ホスト Ollama）」。

## 5. シミュレーション記録：DBは必要か

### 判断

| 要件 | DBあり（SQLite/PostgreSQL等） | **ファイルのみ（JSONL 推奨）** |
|------|-------------------------------|--------------------------------|
| 提出用デモ・数日スケール | セットアップ・マイグレーションのコストが勝ちやすい | **実装が最小・バックアップが容易** |
| 複数環境で集計・検索 | 向いている | 行数が少なければ grep / 小スクリプトで足りる |
| 監査・長期保存 | 向いている | 必要になったら後から SQLite に移せばよい |

**提案（最適解・短期）:**  
**DBサーバーは立てない。** 各実行のサマリを **1行1JSONの JSONL**（例：リポジトリ直下 **`output/demo_runs.jsonl`**）へ追記する。

**SQLite を足すのがよい唯一のケース:** 「提出後も実験を続け、ダッシュボードで集計したい」が既に確定している場合。その場合も **単一ファイル SQLite** で十分で、PostgreSQL などは不要。

### 記録する項目（最小）

- `timestamp`, `scenario_id`, `branch`（continue / llm_stop）  
- 最終ステップの `R_obj`, `R_subj`, `Gap`, `outcome`（accident / avoided）  
- （任意）LLMに渡したプロンプトのハッシュや応答テキスト

- **`apps/demo_web`**：設計書どおりの **判断構造の説明責任** を果たす本体デモ。
