# デモ RP：UI（フラグ常時表示）・多段判断シミュ・LLM チャット拡張（vNext 要件と実装方針）

本書は、現行 `apps/demo_web`（`high_level_design_v2.md` / `rp_simulation.md` と整合）に対し、**体験面とシミュレーション面の拡張要件**を整理する。数値のソース・オブ・トゥルースは引き続き Python（`risk_simulator.RiskSimulator`）を正とし、LLM は**発話・説明・補助判断**に限定する方針は維持する。

---

## 1. 背景（現状の課題）

| 領域 | 現状 | 課題 |
|------|------|------|
| **フラグ表示** | 「客観リスク内訳」内のタブ **「フラグ」** に `続行ルール成立` / `Gap ≥ 0.2` がある（`static/js/demo/metrics-breakdown.js`） | 重要な合否がタブ内に隠れ、常時可視性が低い |
| **最終判断** | `step >= max_steps` のときだけ `decide_continue` / `decide_llm_stop` が有効。選択後に即 `phase=ended` | **続行・中止＝即シナリオ終了**で、最大 70 ステップまでの「通し」や、判断後の揺り戻しが表現しづらい |
| **続行の帰結** | `decide_continue()` は `gap_danger` や `R_obj` 閾値で **事故 / クリアを決定論的に** 分岐 | 閾値超えでも**必ず失敗しない**（幸運・余裕・微修正で持ち直す）ケースを入れたい |
| **パーティチャット** | 引率 ON 時、引率者発言は Ollama の `trek/rest` 用 JSON プロンプト由来。隊員は `_party_member_lines` の**条件分岐テンプレ**（`app.py`） | 会話が型にはまり、引率が**常時「LLM 介入のサポート文」**としての距離感とズレる |

## 2. UI：フラグを「常時見える」場所に置く

### 2.1 推奨配置（第一候補）

**左カラム「主要指標」ブロックの直下**に、高さの小さい **「状態フラグ」カード（またはインライン行）** を置く。

- **理由**
  - 「主要指標」と同じ視線高度にあり、ダッシュボードとしての **読み順（上→下）** が自然。
  - 中央のグラフ・登山シーンを狭めず、左の「数値と状態」列に意味を集約できる。
  - 狭い画面では左カラムが折りたたまれるが、主要指標とセットで最初に読まれる領域のまま。

表示内容（現行ロジックと同一でよい）：

- **続行ルール成立** … はい / いいえ（`metrics.continue_rule_holds`）
- **Gap ≥ 0.2** … はい / いいえ（`metrics.gap_danger`）

必要なら **バッジ色**（例：Gap 危険＝警告色）で一目化する。

### 2.2 代替配置（第二候補）

**ツールバー右側**（`step` 表示の近く）に **コンパクトな 2 つのピル／アイコン** として常時表示。

- **利点**: どのタブやスクロール位置でも見える。
- **欠点**: 文言が細かい場合は折り返しやツールチップが必要。

### 2.3 タブ「フラグ」の扱い

フラグを上記いずれかに移したら、内訳エリアのタブは **環境 / 人 / 圧力の 3 タブ** に整理し、**「フラグ」タブは削除**して重複をなくす（実装時は `renderBreakdownDetail` からフラグ pane を除去）。

## 3. シミュレーション：多段・非決定論・最大ステップまでの通し

### 3.1 要件の整理（プロダクト言語）

1. **人格・状況により**、閾値の効き方や「判断が発動するタイミング」が変わってよい（シナリオ多様性）。
2. ある閾値で **「判断イベント」** が起きても、**続行を選んでも即事故とは限らない**（揺り戻し・小康・確率的成功）。
3. **続行 / 中止** が押されたからといって**必ずシミュレーション全体が終了しない**。問題なければ **設定上限（最大 70 ステップ）** まで進行できる。
4. **続行してメトリクスが悪化**しても、**必ず失敗確定**にはしない（緊張は上げつつ、単調な「危険＝死」ではなく変則的な分岐）。

### 3.2 設計上の置き方（ストーリーとコードの分離）

- **ナラティブ層**（`rp_zone`、背景、チャット）と **数値層**（`R_obj` / `R_subj` / `Gap`）は分離しつつ、**判断イベント**は数値層のフラグで駆動する。
- **乱数**は **シード付き**（再現性のためデモ・ログ比較に必須）。UI からシードを変えられると検証に有利。

### 3.3 状態機械の拡張（提案）

現行：`running` →（上限到達で判断）→ `ended`。

**vNext 案**（概念）：

| 概念 | 説明 |
|------|------|
| `running` | 通常進行。`step < max_steps` の間は `advance` 可能。 |
| `judgment_pending`（任意） | 閾値を跨いだが「まだ確定していない」インタラクション待ち。またはモーダルなしで **自動演出のみ** にもできる。 |
| `paused_after_choice`（任意） | ユーザーが続行/中止を選んだ直後、**即 `ended` にしない**。代わりに「方針フラグ」だけ更新し `running` に戻る。 |
| `ended` | エンディング確定（従来の `outcome` / `rp_zone`）。 |

**「判断」トリガ**の例（複数併用可）：

- **定点**: `step` が特定値、または `step / max_steps` の比率。
- **メトリクス**: `Gap ≥ 0.2`、`R_obj` 帯、合成スコアなど（人格プリセットで閾値オフセット）。
- **確率**: 上記を満たしても毎回ではなく「抽選で判断イベント」→ 同一閾値でも再現性はシードで担保。

**アウトカム**の例：

- 続行選択後：`accident` を **即決しない**。まず `stress_counter` や `luck` バッファを更新し、**後続ステップ**で確率／二次閾値により決着。
- 中止選択後：即 `avoided` で終了せず、「下山シーケンス」として **あと N ステップ** は `advance` 可能にする、など。

※具体的な式は `risk_simulator.py` のチューニングで調整。本書は**振る舞い要件**の固定のみとする。

### 3.4 API / スナップショットに載せるとよい項目（案）

- `judgment_events`: 過去の判断履歴（ステップ、理由コード、選択）。
- `rng_seed` / `run_id`
- `persona_id` およびシミュ用オフセット（閾値・確率係数）

フロントは **「最終判断」ボタンをいつ有効にするか** を `can_decide` から **`decision_mode`**（例：`none` | `soft_hint` | `hard_choice`）へ拡張してもよい。

## 4. パーティチャット：LLM による自由会話と「引率」の役割定義

### 4.1 目標イメージ

- **隊員**も含め、会話を **テンプレ中心から脱却**し、状況・数値・過去発話に応じた **可変ダイアログ** にする。
- **引率者（朔）**は、ロールプレイ上の「現場の一人」だけでなく、デモ文脈では **「LLM 介入時のサポート／メタ的な注意喚起」** に寄せてよい（ユーザーが「システムからの支援メッセージ」として読めるトーン）。

### 4.2 実装の柱

1. **単一ステップあたりの LLM 呼び出し設計**
   - 現行：`trek/rest` + `reasoning` の JSON。
   - 拡張：**構造化出力**（例：`action`, `leader_line`, `member_a_line`, `member_b_line`）または **2 段**（先に action のみ、次に会話のみ）を検討。  
   - **数値の捏造禁止**はプロンプトとサーバ側検証で維持。

2. **履歴の渡し方**
   - `guide_chat` の直近 N ターンをプロンプトに含める。
   - トークン削減のため要約フィールド（サーバ保持）を後から足してもよい。

3. **フォールバック**
   - Ollama 失敗時は現行の `_party_member_lines` / `_leader_fallback_line` に退行（体験が止まらないように）。

4. **UI**
   - 「LLM メッセージ（中止時）」だけでなく、**毎ステップのパーティログが生成主体**であることが分かるラベル調整（`index.html` のサブタイトル等）。

## 5. 実装方針（フェーズ分割）

### フェーズ A — UI（フラグ常時表示）

- `index.html`：左カラムに `#flag-strip` 等を追加、または `app.js` で主要指標の後に `renderFlags(m)` を挿入。
- `metrics-breakdown.js`：フラグタブ削除、スタイルは `demo.css` に最小追加。
- 受け入れ条件：**ページ読み込み直後、タブ操作なしで** 2 フラグが読める。

### フェーズ B — シミュレータ（判断・終了条件）

- `RiskSimulator` に **判断イベント層** と **シード付き RNG** を追加。
- `decide_continue` / `decide_llm_stop` の意味を「即終了」から要件どおりに変更（後方互換が必要ならクエリパラメータ `legacy_decision=true` 等は検討）。
- `rp_simulation.md` のフロー図を、新しい分岐に合わせて更新。

### フェーズ C — LLM チャット

- `app.py`：`_party_chat_entries` を「LLM 生成優先」に差し替え、パース失敗時のみテンプレ。
- プロンプト：引率を **サポート／介入** トーンに寄せる指示を明示。
- ログ：`demo_runs.jsonl` に会話要約またはハッシュを追加するかは運用で決定。
