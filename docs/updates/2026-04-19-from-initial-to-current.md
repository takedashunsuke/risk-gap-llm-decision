# 初期状態からの変更の経緯（2026-04-19 前後）

「いまのリポジトリの挙動」に至るまで、**何を課題にして、何をどう直したか**の流れです。障害の手当て内容の列挙は `../debug/2026-04-19-docker-ollama.md` に分離しています。

---

## 想定していた初期状態（変更前のベース）

- **`main.py`**: 実行前に出力ディレクトリを **`shutil.rmtree` で丸ごと削除**。ホスト単体運用では問題になりにくいが、Compose の **バインドマウント**ではマウント点の削除ができずエラーになり得た。
- **`ollama_client`**: HTTP クライアント側の読み取りタイムアウトが実質 **短め** で、Docker 内 CPU の長い生成と噛み合わないことがあった。
- **`docker/config.docker.yaml`**（過去の経緯では別レイアウトもあった）: ホストより **重め** のシミュ規模・LLM 設定の意図があり、Docker + CPU では **1 ステップあたり・全体**の所要時間が非常に長く、`output` が空に見える・タイムアウトが出るといった体感につながりやすかった。
- **`docker-compose.yml`**: `ollama` と `simulation` の **起動順・ヘルスチェック** はある一方、モデルは **コンテナ側で明示的に pull** しないと、`config` と実物がずれることがあった。
- **ドキュメント**: 運用上のつまずきは口頭・チャットベースになりやすく、後から追いにくかった。

---

## 変更が入った順の流れ（要約）

1. **出力ディレクトリ削除（Docker）**  
   `output` をバインドマウントしているとき、`rmtree('output')` が **EBUSY** で落ちうることを確認した。対応として **`clear_output_directory()`** で「中身だけ削除」を試す方式に変更した。

2. **Ollama との通信・タイムアウト**  
   `Read timed out` やサーバ側の **`POST /api/generate` が長時間／500** と、クライアント側タイムアウトの不一致が問題になった。  
   **`request_timeout` を設定から渡す**、`docker/config.docker.yaml` で **長めの秒数**、`max_tokens` の見直し、必要に応じて **`num_ctx` の縮小** を入れた。

3. **観測性（ログ）**  
   どこで時間がかかっているか切り分けやすくするため、**ステップ境界**・**LLM 呼び出しの開始／成功所要時間**・**HTTP エラー時の本文スニペット**・起動時の **モデル名・主要パラメータ表示**などを追加した。

4. **モデルの所在の取り違え**  
   ホストの Ollama と **Docker 内 Ollama** でモデルストレージが別であることが原因で、「pull したのに not found」が起きた。  
   **`main.py` のモデルチェック失敗時に、コンテナ向けの pull コマンド例をログに出す**ようにした。

5. **Docker 向けデフォルト負荷の調整**  
   CPU・コンテナ前提で現実的に完走しやすいよう、`docker/config.docker.yaml` の **`duration` / `num_agents` / `max_tokens`**、および **`fires.start_step` と `duration` の整合** を段階的に調整した（軽いモデル名の明示など）。

6. **ドキュメントの置き場**  
   つまずきメモを **`docs/debug`** に置いたあと **`docs/updates` に一本化**したが、役割が混ざったため **再編**した。  
   - **`docs/updates`**（本ファイル）: **初期からどう変えてきたか**の経緯  
   - **`docs/debug`**: **いまのトラブルシュート用メモ**（現象・原因・対処の参照用）
